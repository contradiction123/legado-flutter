# Phase 2：书源与规则系统 — 优化执行计划

> **版本**：2.0（基于实际代码状态重新排定）
> **总工期评估**：5 个子阶段，约 5 周
> **核心目标**：完整移植书源规则引擎，实现与原版书源 90% 兼容
> **执行策略**：从 Phase 2.0 依次推进，每阶段产出具可运行代码

---

## 背景

本计划基于以下前提制定：

- **代码现状**：经过实地检查，原文档中标记 ✅ 的内容（AnalyzeRule、WebBook 引擎、书源管理 UI、调试器、换源）均尚未实现。`lib/engine/analyze_rule/`、`lib/engine/web_book/`、`lib/features/` 目录不存在或为空。
- **JS 引擎**：`lib/engine/js_engine/` 已存在基础框架，但 `java.xxx()` 扩展函数全部是"fire-and-forget"模式（通过 `onMessage` 单向通信），JS 端永远收不到返回值。
- **数据库**：35 张表已定义（drift），但 DAO 层和 Repository 层均未创建。
- **领域模型**：仅有 8 个规则模型（freezed），缺 BookSource、Book、BookChapter 等 20+ 个核心模型。
- **依赖**：pubspec.yaml 中缺少规则引擎所需的 `html`、`xpath_selector` 等关键包。

---

## 架构决定

| 决策点 | 方案 | 理由 |
|--------|------|------|
| JS 桥接层 | **Promise + Callback ID 模式** | 解决 `onMessage` 单向通信问题，JS 端可获得返回值 |
| XPath 实现 | **纯 Dart：`xpath_selector` + `xpath_selector_html_parser`** | 零原生代码、全平台兼容，免去 FFI/libxml2 的集成复杂度 |
| 数据访问层 | **DAO + Repository 两层模式** | DAO 封装查询逻辑，Repository 处理 `Table ↔ Model` 映射 |
| 领域模型 | **独立 freezed 模型 + 映射器** | 模型与数据库解耦，可单独测试和复用 |
| JS 扩展函数 | **全部 50 个实现返回值** | 确保与原版 Rhino API 完全兼容 |
| 执行顺序 | **从 Phase 2.0 依次推进** | 根除依赖隐患，稳步推进 |

### 2.1.1 JS 桥接层重构（Promise + Callback ID）

当前 `bridge.js` 中 `java.xxx()` 的定义模式：

```javascript
java.md5Encode = function(str) {
  sendMessage('java.md5Encode', [str]);
  return ''; // ← 永远返回空字符串
};
```

改为 Promise 模式：

```javascript
// 全局回调注册表
var _callbacks = {};
var _callbackId = 0;

// 异步调用桥接
function _callBridge(channel, args) {
  return new Promise(function(resolve, reject) {
    var id = ++_callbackId;
    _callbacks[id] = { resolve: resolve, reject: reject };
    sendMessage(channel, { callbackId: id, args: args });
  });
}

// 接收 Dart 端回调结果
function _onBridgeResult(callbackId, error, result) {
  var cb = _callbacks[callbackId];
  if (!cb) return;
  delete _callbacks[callbackId];
  if (error) cb.reject(new Error(error));
  else cb.resolve(result);
}

// 改造所有 java.xxx() 为异步
java.md5Encode = function(str) {
  return _callBridge('java.md5Encode', [str]);
};
```

Dart 端新增 `_sendResult(callbackId, result)` 方法：

```dart
// JS 引擎封装新增
void sendResult(int callbackId, dynamic result) {
  evaluate('_onBridgeResult($callbackId, null, ${jsonEncode(result)})');
}
```

import 'package:xpath_selector_html_parser/xpath_selector_html_parser.dart';
import 'package:html/parser.dart' as html;

class AnalyzeByXPath {
  List<String> getStrings(String htmlContent, String xpathExpr) {
    final doc = html.parse(htmlContent);
    final result = doc.queryXPath(xpathExpr);
    return result.nodes.map((n) => n.stringValue ?? '').toList();
  }
}
```

  Future<List<SearchBook>> explore(BookSource source, String? url);
}
```

核心约定：

1. **searchBooks**：从书源取 `searchUrl` → `AnalyzeUrl` 拼接 → HTTP 请求 → `AnalyzeRule` 解析 `ruleSearch` → 返回 `List<SearchBook>`
2. **getBookInfo**：请求 `bookUrl` → `AnalyzeRule` 解析 `ruleBookInfo` → 返回填充后的 `Book`
3. **getChapterList**：请求 `tocUrl` → `AnalyzeRule` 解析 `ruleToc` → 返回 `List<BookChapter>`
4. **getBookContent**：请求章节 URL → `AnalyzeRule` 解析 `ruleContent` → 返回正文
5. **explore**：使用 `exploreUrl` 模板 → 解析 `ruleExplore` → 返回发现列表

并发策略：

- 多书源搜索时，使用 `Future.wait()` + `Stream` 渐进展示已有结果
- 单书源内部请求按依赖链串行（搜索→详情→目录→正文）
- 可配置并发率（`concurrentRate` 字段）

## 子阶段 2.4：集成测试与兼容性验证

**工期评估**：5 天  
**前置依赖**：2.3（全部功能就绪）  
**产出**：书源 90% 兼容性得到验证，修复发现的兼容性问题

### 2.4.1 测试书源准备

- 用户提供书源 .json 文件
- 导入到应用的测试环境
- 按网站类型/难度分级：

| 级别 | 说明 | 典型书源数 |
|------|------|-----------|
| P0 - 主流网站 | 笔趣阁、妙笔阁、69书吧等 | 20-30 |
| P1 - 常见网站 | 番茄、起点、晋江等 | 30-50 |
| P2 - 小众/反爬网站 | 有 TTF、特殊编码等 | 20-30 |
| P3 - 边缘情况 | 非常规规则、纯 JS 渲染 | 10-20 |

### 2.4.2 兼容性测试矩阵

对每个书源测试 5 个核心流程：

| 流程 | 输入 | 期望 | 实测 |
|------|------|------|------|
| 搜索 | 关键词（如"凡人修仙传"） | 返回结果列表，含书名/作者/封面 | |
| 详情 | 点击搜索结果 | 展示完整书籍信息 | |
| 目录 | 进入目录页 | 展示章节列表，章节数 > 10 | |
| 正文 | 点击任一章节 | 展示正文内容 > 100 字 | |
| 发现 | 使用发现 URL | 返回发现列表 | |

### 2.4.3 兼容性问题分类与修复

| 问题类型 | 常见原因 | 修复策略 | 优先级 |
|---------|---------|---------|-------|
| CSS 选择器不匹配 | html 包选择器与 jsoup 差异 | 添加适配器层或自定义选择器 | P0 |
| XPath 语法不支持 | xpath_selector 的功能边界 | 给 xpath_selector 提 PR 或添加 polyfill | P0 |
| JS 扩展函数名/参数不匹配 | 书源使用了非标准 API | 分析原版 Rhino 源码补全 | P0 |
| 编码问题 | 响应编码检测不准确 | 增加 charset 检测逻辑 | P1 |
| Cookie 问题 | CookieJar 未正确配置 | 修复 Cookie 管理器 | P1 |
| TTF 字体反爬 | 原版 QueryTTF 逻辑 | 实现简单的 TTF 解析（字体映射表） | P2 |

### 2.4.4 最终兼容率统计

```
兼容率 = (书源通过流程数 / 书源总数 × 5) × 100%

报告格式：
├── P0 书源通过率: XX/XX (XX%)
├── P1 书源通过率: XX/XX (XX%)
├── P2 书源通过率: XX/XX (XX%)
└── 总体兼容率: XX/XX (XX%)
```

| 指标 | 目标 |
|------|------|
| 单书源搜索响应时间 | ≤ 3s（网络正常时） |
| 10 书源并发搜索完成时间 | ≤ 8s |

## 风险登记册

| 风险 | 等级 | 概率 | 影响 | 应对方案 |
|------|------|------|------|---------|
| xpath_selector 对复杂 XPath 支持不足 | 🔴 | 中 | 高 | Plan A: 给 xpath_selector 提 PR 补充；Plan B: 对不支持的功能 fallback 到 WebView 执行 `document.evaluate()` |
| flutter_js (QuickJS) 的异步 Promise 桥接不支持 | 🔴 | 中 | 高 | Plan A: 用 `Completer` + 超时机制模拟同步；Plan B: 换用 `mozilla_iot_js_sdk` 或 `flutter_vm` |
| pointycastle AES 加密参数与 Java 不兼容 | 🟡 | 中 | 中 | 需要对齐原版 Java 的 AES/CBC/PKCS5Padding 的具体参数（iv 处理、key 长度） |
| 大量书源的 JS 库（jsLib）使用了 Rhino 特有 API | 🟡 | 高 | 高 | 分析常见书源的 jsLib 模式，逐一提供 Dart polyfill |
| 7z/RAR 解压在纯 Dart 生态无成熟库 | 🟡 | 高 | 低 | 使用频率低，stub 返回错误不影响核心流程；如需支持，通过 FFI 调用系统命令 |
| TTF 字体反爬解析复杂 | 🟡 | 中 | 低 | 先用字体映射表硬编码常见反爬字体，后续可集成 opentype.js |
| 未见过的书源规则格式导致解析异常 | 🟡 | 中 | 中 | 调试器提供"查看原始规则"功能 + 友好错误提示 |
| 多书源并发搜索导致内存/连接池耗尽 | 🟡 | 中 | 中 | 限制最大并发数（默认 10），使用 Semaphore 控制 |
